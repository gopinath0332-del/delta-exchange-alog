## 🔴 High Priority — Accuracy / Correctness Issues

### 1. Slippage Modeling (`engine.py`)

**Problem**: Trades execute _exactly_ at the signal price with zero slippage. In reality, Delta Exchange market orders fill at a slightly worse price due to spread and queue depth.

**Fix**: Add a configurable `slippage_pct` (e.g. `0.05%`) in `settings.yaml` under `backtesting:`.  
Apply it in `engine.py → run()`:
- Long entries: `entry_price *= (1 + slippage_pct)`
- Short entries: `entry_price *= (1 - slippage_pct)`
- Exits: reverse direction

**Location**: `backtest/engine.py` → `run()` loop, lines ~95–107

---

### 2. Funding Rate Costs Not Modeled (`engine.py`)

**Problem**: Perpetual futures charge funding fees every 8 hours. The current backtest ignores this entirely, causing PnL to be **overstated** for long-held positions in positive funding environments.

**Fix**: Add a `funding_rate_per_8h` config key (e.g. `0.01%`). Compute deduction:
```
funding_cost = (bars_held * bar_duration_hours / 8) * funding_rate * notional
pnl -= funding_cost
```

**Location**: `backtest/engine.py` → after PnL calculation, using `bars_held`

---

### 3. Intra-Bar Signal Priority Ambiguity (`donchian_strategy.py`)

**Problem**: In the Donchian backtest loop (line 652–655), the partial TP checks `high >= tp_level` and the trailing stop checks `close <= trailing_stop`. Within the same bar, it is unknown whether the high came before or after the close, making signal ordering ambiguous across strategies.

**Fix**: Establish a **strict priority order** within each bar for all strategies:
1. Partial TP hit (checked against `high`/`low`)
2. Trailing stop hit (checked against `close`)
3. Channel exit (checked against `close`)
4. New entry (only if flat)

Enforce this ordering consistently in `run_backtest()` implementations.

**Location**: `strategies/donchian_strategy.py` → `run_backtest()` loop, lines ~628–714

---

### 4. Sharpe Ratio Fallback Is Not Annualized (`metrics.py`)

**Problem**: When daily resampling fails (e.g. dates can't be parsed), the code falls back to per-trade Sharpe (lines 114–122) **without annualizing**, making it incomparable to the daily-resampled version.

**Current (incorrect)**:
```python
sharpe_ratio = (returns.mean() / returns.std())
```

**Fix**: Estimate annualization factor from average trade frequency:
```python
# Approximate annualization: assume ~252 trading days, estimate trades/day
trades_per_year = (num_trades / data_period_days) * 252  # if data_period_days known
sharpe_ratio = (returns.mean() / returns.std()) * np.sqrt(trades_per_year)
```

**Location**: `backtest/metrics.py` → lines ~114–122

## 🟡 Medium Priority — Useful New Features

### 5. Walk-Forward / Out-of-Sample (OOS) Testing (`run_backtest.py`)

**Problem**: The entire dataset is used for testing (in-sample only). This can lead to overfitting — a strategy may look great on historical data but fail live.

**Fix**: Add CLI arguments `--split-date <YYYY-MM-DD>` or `--oos-pct 0.3` to `run_backtest.py`.  
Split data into:
- **In-Sample (IS)**: used for reviewing / tuning
- **Out-of-Sample (OOS)**: held-out period, reported separately  

Generate two separate metric blocks and report both in the HTML.

**Location**: `run_backtest.py` → `main()` and `run_backtest_for_file()`

---

### 6. Parameter Grid Optimization (`run_backtest.py`)

**Problem**: To try different strategy parameters (e.g. `enter_period: 10 vs 20 vs 30`), you must manually edit `settings.yaml` and re-run.

**Fix**: Add `--optimize` mode. Read a parameter grid from a new `backtesting.optimize` section in `settings.yaml`:
```yaml
backtesting:
  optimize:
    donchian_channel:
      enter_period: [10, 20, 30]
      atr_mult_tp: [2.0, 3.0, 4.0]
```
Run all combinations, rank by Sharpe or Profit Factor, export results to `reports/optimization_results.csv`.

**Location**: New function in `run_backtest.py` + strategy factory modifications

---

### 7. Monte Carlo Simulation (`metrics.py`)

**Problem**: The current metrics represent a single deterministic equity path. In reality, trade sequence has randomness — a string of early losses could blow up equity even if the strategy is profitable overall.

**Fix**: After backtest, shuffle trade sequence N times (e.g. 1000) and compute:
- **5th percentile max drawdown** (stress-test)
- **5th percentile final return**
- **Probability of ruin** (equity < 50% of initial)

Add `monte_carlo_simulations: 1000` to `settings.yaml` under `backtesting:`.

**Location**: `backtest/metrics.py` → new `run_monte_carlo()` function called from `run_backtest.py`

---

### 8. Live Data Fetch for Backtest (`data_loader.py` + `run_backtest.py`)

**Problem**: You must manually export CSVs from TradingView or other sources before running a backtest. There's no way to fetch data directly from Delta Exchange.

**Fix**: Add `--fetch-live` flag to `run_backtest.py`. When set:
1. Call `api/rest_client.py` to download historical candles via `/v2/history/candles`
2. Save to configured `data_folder` as `{SYMBOL}_{TIMEFRAME}.csv`
3. Proceed with backtest automatically

**Location**: `run_backtest.py` → `main()`, new `DataFetcher` class in `data/fetcher.py`

---

### 9. Multi-Strategy Comparison Report (`reporter.py`)

**Problem**: Each strategy/symbol gets its own HTML. When running multiple strategies in batch, there's no unified comparison page.

**Fix**: After all backtests complete in `run_backtest.py`, generate one **comparison HTML** with:
- Sortable table (already partially done in terminal output)
- Combined overlaid equity curves on a single Plotly chart
- Highlight the best-performing config

**Location**: `backtest/reporter.py` → new `generate_comparison_report()` method + call in `run_backtest.py` `main()`

## 🟢 Low Priority — Polish & Quality-of-Life

### 10. Deprecated `fillna(method='ffill')` (`data_loader.py`)

**Problem**: Line 98 uses `df.fillna(method='ffill')` which is deprecated in pandas ≥ 2.0.

**Fix**:
```python
# Old (deprecated):
df = df.fillna(method='ffill')
# New:
df = df.ffill()
```

**Location**: `backtest/data_loader.py` → line 98

### 12. Reports Deleted on Every Run (`reporter.py`)

**Problem**: `Reporter.__init__()` deletes all existing `.html` and `.csv` files in `reports/` on startup. If a batch of 8 strategies is partially run and fails, all prior reports are lost.

**Fix**: Write to a **timestamped subdirectory** instead of clearing:
```python
# Instead of clearing, use:
timestamp = datetime.datetime.now().strftime("%Y-%m-%d_%H-%M")
self.reports_dir = Path(reports_dir) / timestamp
self.reports_dir.mkdir(parents=True, exist_ok=True)
```

**Location**: `backtest/reporter.py` → `__init__()`, lines ~16–27

### 13. No Base Strategy Class (`strategies/`)

**Problem**: Every strategy duplicates `trades`, `active_trade`, `current_position`, `run_backtest()` boilerplate. This makes maintenance harder and means the engine cannot be truly strategy-agnostic.

**Fix**: Create `strategies/base.py` with abstract methods:
- `check_signals(df, current_time_ms, **kwargs) -> (action, reason)`
- `update_position_state(action, ...)`
- `run_backtest(df)`
- `reconcile_position(size, entry_price)`

Also move `trades`, `active_trade`, `current_position` state management into the base class.

**Location**: New file `strategies/base.py`, then update all existing strategies to inherit from it

### 14. Missing Performance Metrics (`metrics.py`)

**Problem**: Several standard metrics present in TradingView's strategy tester are missing:

| Metric | Formula | Importance |
|--------|---------|------------|
| **CAGR** | `(final/initial)^(1/years) - 1` | Normalized annual return |
| **Recovery Factor** | `Total Return / Max Drawdown` | How quickly equity recovers |
| **Calmar Ratio** | `CAGR / Max Drawdown%` | Risk-adjusted return measure |
| **Consecutive Wins** | Max streak of winning trades | Streak risk |
| **Consecutive Losses** | Max streak of losing trades | Streak risk |
| **Avg Trade Duration (days)** | `mean(exit_time - entry_time)` in days | Trade holding analysis |

**Fix**: Add calculations in `backtest/metrics.py` → `calculate_metrics()` and expose in `Detailed Table` + terminal summary.

**Location**: `backtrack/metrics.py` → `calculate_metrics()` function

### 15. MAE / MFE Per-Trade Metrics (`engine.py`, `metrics.py`, `reporter.py`) ✅ IMPLEMENTED

**Problem**: No per-trade analysis of how much profit was left on the table (MFE) or how much heat a trade took before resolving (MAE).

**Fix Implemented**:
- `backtest/engine.py` → New `_calculate_mae_mfe()` helper scans the OHLC candle slice for each trade and attaches `mae_price`, `mae_pct`, `mfe_price`, `mfe_pct` to every `processed_trade` dict.
- `backtest/metrics.py` → `calc_stats()` now aggregates avg/max for both metrics (All / Long / Short). A new **MAE / MFE Analysis** header section is added to the Detailed Table in the HTML report.
- `backtest/templates/report_template.html` → **MAE %** (orange) and **MFE %** (blue) columns added to the Trades table.
- `backtest/reporter.py` → New `_create_mae_mfe_chart()` generates a Plotly scatter plot (X = MFE%, Y = MAE%, green = winners, red = losers, diagonal reference line).
- `tests/unit/test_mae_mfe.py` → 14 unit tests (LONG, SHORT, single-bar, invalid timestamps, processed_trade key smoke-test) — all passing.
- `README.md` → New **MAE / MFE Backtest Metrics** section with interpretation guide.
